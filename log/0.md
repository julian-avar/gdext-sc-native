Roadmap to Fully Featured

Ordered roughly by impact/dependency:

### 1. Full Virtual Dispatch (high impact, mostly mechanical)
The `{Class}Virtuals.scala` files are already generated. `ClassRegistrar.buildGetVirtualFn()` needs to iterate them instead of the 3 hardcoded comparisons. Requires each registered class to carry its virtual table, looked up by StringName intern pointer.

### 2. `@gdclass` Macro — Auto-Registration (high ergonomics)
Currently users hand-write `GdClassRegistry.register(...)` and `Foo.Binds.loadBinds()`. A Scala 3 compile-time macro (or a Mill source generator) should:
- Scan classes annotated `@gdclass`
- Emit the `register(...)` call
- Emit all `Binds.loadBinds()` calls in the scene init callback

### 3. `@export` — Editor Properties (enables real game dev)
Requires:
- `classdb_register_extension_class_property` call per field
- `GDExtensionClassSet/GetFunc` callbacks dispatching to Scala fields via Variant
- Variant marshaling layer (the hardest part — packing/unpacking `Ptr[Byte]` as Godot Variants)

### 4. Typed Signal Connections + `@signal`
- The `connect(signal)(cb: () => Unit)` API needs to pass Variant arguments through to the closure
- `@signal` on an extension class should register the signal at class-registration time via `classdb_register_extension_class_signal`

### 5. `@func` — Method Export to GDScript
Lets GDScript call Scala methods. Requires Variant marshaling and `classdb_register_extension_class_method`.

### 6. RefCounted Safety
`Gd[T]` for `RefCounted` subtypes should call `ref`/`unref`. The type system already has the variance (`Gd[+T]`); just needs runtime ref-tracking, likely via a wrapper that calls `reference`/`unreference` on assign/drop.

### 7. Signal Disconnect + `@onready`
- `CallbackRegistry` needs a `deregister(id)` path and the signal `disconnect` call
- `@onready` is syntactic sugar: `val label: Label = null` becomes `label = getNode("Label").as(new Label(_))` inserted at the top of `_ready`

### 8. Auto-load of All Binds
Right now every binds object must be manually initialized. The scene-init callback should auto-call `loadBinds()` on every generated class that has been instantiated, or a single generated `AllBinds.loadAll()`.

## Summary Assessment

The project has the hardest parts done: the FFI handshake, class registration, ptrcall dispatch, builtin types, and a working code generator. The example `ExampleSceneScala` already demonstrates a genuinely clean user API.

The main gap between "works for demos" and "fully featured" is the **Variant layer** — without it, `@export`, `@func`, and typed signals can't be implemented. That is the single highest-leverage piece of work remaining. Everything else (virtuals, macros, disconnect) is mechanical once Variant marshaling exists.

Binds: what SwiftGodot and godot-rust do

Both use **lazy initialization** — the bind is loaded on the first call to that method, then cached. No explicit `loadAll()` or `loadBinds()` call is needed anywhere.

**godot-rust:** each method bind is stored in a `static OnceLock<MethodBindPtr>`. The generated method body checks the lock, initializes on first call, then just reads the cached pointer on every subsequent call.

**SwiftGodot:** same idea — a `static var _method_xxx: GDExtensionMethodBindPtr! = nil` guard at the top of each generated method.

The fix for this project is equivalent and actually simpler in Scala 3: switch from `var` to **`lazy val`** in each `Binds` object. After the first call the `lazy val` is just a field read — zero overhead. The `loadBinds()` methods and `AllBinds.loadAll()` idea disappear entirely.

The generated pattern changes from:
```scala
object Binds {
  var addChild: Ptr[Byte] = null   // crashes if loadBinds() forgotten
  def loadBinds(): Unit = {
    addChild = GdxApi.getMethodBind(c"Node", c"add_child", 3070154285L)
  }
}
```
to:
```scala
object Binds {
  lazy val addChild: Ptr[Byte] = GdxApi.getMethodBind(c"Node", c"add_child", 3070154285L)
}
```

That's it — `GdxApi` is already initialized before any user code calls a method (guaranteed by the init callback flow), so the lazy load is always safe.

### RefCounted and the GC

The GC docs are relevant here in a specific way. The Scala Native Immix GC manages **Scala objects** (your `GodotObject` subclass instances, closures, Maps, etc.). The GC has no visibility into Godot's own reference-counting system at all — it just sees `Ptr[Byte]` as an opaque number.

These are two entirely separate ownership systems running in parallel:

```
Scala GC world                  Godot refcount world
─────────────────               ──────────────────────
GodotObject (Scala wrapper)     Godot native object
  var ptr: Ptr[Byte] ──────────→  refcount = 1
  (tracked by GC)                  (tracked by Godot)
```

For `Node` subclasses this is simple: Godot owns them via the scene tree, the refcount never goes to zero while the node is in the tree. Your Scala wrapper just holds a raw pointer and everything is fine.

For `RefCounted` subclasses (`Resource`, `AudioStream`, `Texture`, etc.) it matters: if you receive one from an engine method call, Godot's returned reference is "unowned" — you're supposed to call `reference()` to claim it. If you don't, Godot might free the object while your `Gd[Resource]` still points at it.

**What "GC'd for now" means in practice:**

- The GC manages the lifetime of your Scala wrapper objects — you never call `free()` on them. ✓ (already working)
- For RefCounted engine objects, the pragmatic starting point is: **call `reference()` on every acquire, never call `unreference()`**. The Godot object leaks (refcount stays ≥ 1 forever), but your pointer is always valid. For a typical game (Resources loaded at startup, not created/destroyed in hot loops), this is invisible in practice.
- The GC docs' note about Immix being **non-moving** is actually a key guarantee: captured Scala objects (closures in `CallbackRegistry`, `GodotObject` instances in `instanceMap`) will never be relocated, so the raw pointers Godot holds back into Scala memory remain valid. This is why the current architecture works.

The real fix for RefCounted is: when the generated code returns a `Gd[T]` where `T extends RefCounted`, emit a `reference()` call on the returned object, and expose an explicit `free()` on `Gd[T]` that calls `unreference()` — making ownership explicit. Full drop semantics (like Rust's) aren't possible in Scala/GC without finalizers, which are unreliable. Explicit `free()` is the pragmatic answer.
